## 二、关键设计决策与用户指挥记录

### 1. Token 用量统计

- **问题**：`benchmark_evaluator.py` 中 `total_input_tokens` 和 `total_output_tokens` 硬编码为 0
- **根因追踪**：`handyllm` 的 `evaled.run()` 返回 `ChatPrompt` 对象 → `response` 属性是原始 `ChatResponse` 字典 → `response["usage"]` 包含 `prompt_tokens` / `completion_tokens` / `total_tokens`
- **修复链路**：
  1. `workflow_localization.py`：`_call_llm` 中提取 `response.response["usage"]` 存为 `self._last_usage`
  2. `intent_recognition_v2.py`：同上
  3. `main_feedback.py`：`process_from_dicts` 中收集两个服务的 `_last_usage` 累加后写入 `results["usage"]`
  4. `benchmark_evaluator.py`：从 `result["usage"]` 读取填充
- **验证结果**：E1 (1 次 LLM) 约 6887+894tok；E2 (2 次 LLM) 约 15843+1709tok，正确累加

### 3. E2 意图识别 prompt 重构

- **问题 1**：E2 的 localize_input 中 `%intent_info%` 丢失了任务描述和正确答案
  - **根因**：E2 用 `intent_result.raw_feedback` 作为 base text，但这是 LLM 返回的（只保留了最后一句），不是原始 `feedback_text`
  - **修复**：`process_from_dicts` 把原始 `feedback_text` 传给 `localize()`；E2 模式下用 `feedback_text` 打底
- **问题 2**：意图识别输出 70 行 JSON 噪音，对定位无帮助
  - **用户确认**：角色定位为"轨迹摘要器"，输出结构全新设计
  - **修复**：新建 `intent_recognition_benchmark.hprompt`，输出 4 个精准字段：`task_understanding` / `trace_summary` / `anomalies` / `preliminary_hypothesis`
- **用户反馈**："不需要在 py 里做过多提取"
  - **修正**：`_extract_intent_summary` 简化为直接透传 `processing_notes`（原始 JSON 字符串），不做 Python 层格式化

## 三、犯过的错误

### 错误 1：tqdm 与 loguru 冲突导致进度条不显示
- **现象**：终端输出被 loguru 的 DEBUG 日志淹没，tqdm 进度条被冲掉
- **第一次修复**：`logger.remove()` + 重定向到文件，baseline 内部 tqdm 的 stderr 静音
- **第二次修复**：tqdm 仍有缓冲问题（结果文件已生成但终端无输出），用户要求改用 print
- **最终方案**：全部改为 `print(..., flush=True)`，彻底解决

### 错误 2：E2 丢失任务描述和正确答案
- **现象**：E2 的 localize_input 中 `%intent_info%` 只有"该多 Agent 系统未能给出正确答案..."，没有任务描述和正确答案
- **根因**：`intent_result.raw_feedback` 是 LLM 返回值（LLM 把原文截断了），不是我们构造的 `feedback_text`
- **修复**：在 `localize()` 签名链路上新增 `feedback_text` 参数，E2 模式下用它打底

### 错误 3：采样数据不更新
- **现象**：用户改了 `--n_ag` 和 `--n_hc` 的默认值，但跑出来还是 5+5
- **根因**：`sample_data()` 判断"目录已有数据就跳过"，没检查数量是否匹配
- **修复**：加数量匹配检查；`sampled_data` 移入每个 run 目录

### 错误 4：B1-B6 token 和 LLM 调用次数为 0
- **现象**：效率图中 B1-B6 的 token 全是 0，B2/B3/B5/B6 调用次数也是 0
- **根因**：baseline 方法通过 `Lib/utils.py` 直接调 OpenAI API，没有 usage 回传机制；`llm_calls` 按 method 硬编码（step_by_step/binary_search 写死为 0）
- **修复**：新建 `_TokenTracker` 代理类包装 OpenAI client，拦截 `chat.completions.create()` 自动累计 token 和调用次数

### 错误 5：意图识别 prompt 设计不当
- **现象**：`processing_notes` 偶尔有用信息，但大量输出是 sentiment/FeedbackType/alternative_intents 等噪音
- **根因**：`intent_recognition.hprompt` 是为通用"用户反馈工作流"设计的（"太慢了""改参数"），和 benchmark 场景完全不匹配
- **修复**：新建 `intent_recognition_benchmark.hprompt` 专用 prompt，输出 task_understanding / trace_summary / anomalies / preliminary_hypothesis
