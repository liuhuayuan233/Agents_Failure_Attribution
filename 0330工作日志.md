### 3. 对话历史精简

- **用户指挥**："对话历史里不要有 input 这个键"、"不需要 args_used 和 status"、"不需要顶层 workflow_name 等元数据"
- **问题**：`_build_trace_for_display()` 已生成精简版 trace，但 `process_from_dicts()` 没接收和使用它——仍将完整 trace 传给了 `localize()` 作为 prompt 显示数据
- **修复**：
  - `main_feedback.py`：新增 `execution_trace_raw` 参数
  - `benchmark_evaluator.py` 和 `run_test.py`：传入 `sample.get("execution_trace_raw")`
- **最终精简版 trace 字段**：`step`、`node_name`、`node_type`、`output`（无 inputs/status/args_used，无顶层元数据）

### 4. Step 编号

- **用户指挥**："历史记录里的 step 要从 1 开始计数"
- **实现**：trace 和工作流图都改为 1-indexed，`extract_prediction()` 中做 -1 转换
- **用户反回**："历史记录里 step 还是从 0 开始吧"
- **修正**：全部还原为 0-indexed，撤销 `extract_prediction()` 中的 -1 转换

### 5. Prompt 输出格式精简

- **用户指挥**："不需要 assessment 字段和 target_scope、stage_context 字段，root_cause_analysis 字段也不需要"
- **修改**：
  - Prompt 输出格式从 4 个顶层字段精简为 `error_step` + `ranked_candidates`
  - `_parse_response()` 兼容简化输出：自动从 `ranked_candidates[0]` 派生 target_scope/confidence/summary，构造完整 `LocalizationResult`

### 6. Step/Agent 一致性问题（多轮迭代）

- **问题**：LLM 输出中不同 Agent 共享同一 step 编号（如 Orchestrator 和 WebSurfer 都在 step 16）
- **用户指挥**："一个 step 就对应一个 node，为什么会出现两个呢？在提示词里说明一下"

| 迭代 | 修改 |
|------|------|
| 第 1 轮 | 在 Step 编号规则中增加"ranked_candidates 里不同 Agent 的 step 必须各不相同" |
| 第 2 轮（HC/34 仍重名） | 强化为"每个 candidate 的 step 必须是该 Agent 自己发言的 step，禁止将某个 step 归因给在该 step 没有发言的 Agent" |
| 第 3 轮 | 排名候选规则独立一条"不同候选的 step 不可重复" |

- **最终验证**：HC/34 和 HC/40 两个之前重名的 case 都正确了

### 7. 错误步骤与错误 Agent 的对应关系

- **用户分析**（HC/40 case）：WebSurfer step 16 的消息中包含 Orchestrator 的异常堆栈。数据集标注 mistake_agent=WebSurfer。错误应归因于消息所属的 Agent，而非堆栈中提到的内部组件。

  - 示例替换为 HC/40 真实 case（WebSurfer step 16 含 Orchestrator 异常）
  - 字段说明强化 error_step 与 target_scope 一致性

## 三、犯过的错误

### 错误 1：精简版 trace 没有传入管线

- **现象**：`_build_trace_for_display()` 已生成精简版 trace，但 prompt 中仍显示完整 trace（含 inputs/status/args_used 和顶层元数据）
- **根因**：`adapt_benchmark_sample()` 把精简版存在 `execution_trace_raw` 字段，但 `process_from_dicts()` 没有接收该参数——它将完整 `execution_trace` dict 直接传给了 `localize()` 的 `execution_trace_raw`
- **修复**：`process_from_dicts()` 新增 `execution_trace_raw` 参数，`benchmark_evaluator.py` 和 `run_test.py` 传入 `sample.get("execution_trace_raw")`

### 错误 2：Step 1-indexed 来回折腾

- **经过**：用户要求 step 从 1 开始 → 实现了全链路 1-indexed + extract_prediction 中 -1 转换 → 用户改主意"还是从 0 开始吧" → 全部还原
- **教训**：涉及编号系统的改动影响面广（trace/workflow_graph/prompt 示例/评测比较），应先确认再改

### 错误 3：工作流架构 agents_count 反复调整

- **经过**：初版包含所有节点 → 用户指出 ExecutionNode 不属于 agent → 排除了 → 用户又要求 HumanInputNode 和 ExecutionNode 进入架构 → 恢复所有节点，字段改名为 `nodes_count`
- **教训**：字段语义要明确，`agents_count` 命名暗示只统计 Agent，改为 `nodes_count` 更准确

### 错误 4：Step/Agent 一致性约束不够强

- **经过**：第一轮添加"不同 Agent 的 step 不可相同"，但 LLM 仍违反（HC/34 case）→ 第二轮强化为"step 必须是该 Agent 自己发言的 step" + 具体示例 → 最终修复
- **教训**：对 LLM 的行为约束需要多角度冗余表述，仅说"不可重复"不够，需要解释为什么不可重复并给出正例
