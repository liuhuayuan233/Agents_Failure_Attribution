### 3. 提示词迭代（多轮实验）

#### 基线对比

|------|--------|-----------|-------------|-------------|

#### v1 退步分析（45% vs 70%）

**用户指挥**："对比下两种结果，告诉我两者判断的diff"

**分析结论**：9个样本变化，7退步2进步。退步模式一致——新版选择了错误"显现/确认"的下游步骤而非"引入"的上游根因步骤：
- AG/124：旧版归因给编写bug代码的 Research_Expert(s8)，新版归因给执行报错的 Computer_terminal(s9)
- AG/70：旧版归因给误解任务的 PythonDebugging_Expert(s0)，新版归因给确认错误结论的 Validation_Expert(s2)

**根因**：去掉 `assessment` + `root_cause_analysis` 结构化输出后，模型失去了从症状到根因的推理脚手架。

#### v2 修复尝试

**用户指挥**："综合二者再尝试一次。step是不能放宽的，也不想引入额外字段。但可以尝试在提示词里告诉模型要从症状追溯到根因"

**修改**：
1. 新增"从症状追溯根因"核心原则章节（区分症状 vs 根因，明确目标是定位根因）
2. 修改"错误步骤与错误 Agent 的对应关系"：从"指向错误实际出现的 step"改为"指向引入错误的根因 Agent 发言的 step"
3. 新增示例1（代码执行失败→归因于代码提交者），保留示例2（系统异常）
4. 分析流程建议中增加"向前追溯根因"步骤

**效果**：E1 从 45% 恢复到 65%，v1 退步的 4 个样本有 4 个被修复，零退步。

#### v3 数据层改动（效果分裂，已回退）

**用户指挥**："execution_chain 不论有多长都应该完整还原；outputs 不希望出现'共xx次转换'。不需要做逻辑变化，忠实还原"

**修改**：
- `_content_preview` 截断从 1000 缩短到 300
- 删除 `MAX_EXAMPLES` 截断和"共N次转换"逻辑
- 删除 `_compress_execution_chain`，直接完整输出

**效果**：E2 受益（+10% Agent Acc），E1 受损（-15%）。更多细节反而干扰了 LLM 对短对话样本的判断——退步样本的 LLM 输出明确表示"虽然它是更早步骤，但真正造成确认错误的关键在后续验证结论"。

#### v4 尝试修复 v3（部分改善）

**用户指挥**："再简短工作流架构中的output的截断。然后在提示词最后再强调找根因"

**修改**：`_content_preview` 截断改为 120；提示词末尾加"再次提醒：找最早引入错误的根因 Agent"

**效果**：AG/124 被修复（Computer_terminal→Research_Expert），E2 大幅进步（+20%），但 E1 仍比 v2 差（60% vs 65%）。

#### 最终决定：全部回退到 v2

**用户指挥**：

## 三、犯过的错误

### 错误 1：代码层硬过滤 ranked_candidates

**现象**：用户说"从代码层面把Computer_terminal加入到排除"，我理解为在代码中过滤 LLM 返回的 ranked_candidates
**用户纠正**：不该在代码层面做过滤，应该通过提示词约束
**教训**：用户说"代码层面"是指数据准备层面（node_type 分类），不是结果后处理层面

### 错误 2：工作流架构完整还原导致 E1 退步

**现象**：删除 `_compress_execution_chain` 和 `MAX_EXAMPLES` 截断后，E1 Agent Acc 从 65% 降到 50%
**根因**：更长的输入增加了噪音，LLM 在短对话样本上判断漂移，倾向于选择"确认错误"的下游 Agent 而非"引入错误"的上游 Agent
**修正**：回退到截断版本

### 错误 3：提示词中显式提到 Computer_terminal 不够谨慎

**现象**：在排名规则中展开列出 Computer_terminal、ComputerTerminal 等具体名字
**用户指挥**：回退提示词到 v2 版本（不含这些显式列举）

### 错误 4：连续迭代中未及时确认方向

多次修改中（v2→v3→v4→回退），每次改动同时涉及提示词和数据层，导致效果难以归因。应该每次只改一个变量。

### 当前提示词版本（v2，run_0331_1523）的核心特征

1. **精简输出格式**：仅 `error_step` + `ranked_candidates`（无 assessment/root_cause_analysis）
2. **"从症状追溯根因"原则**：明确区分症状 vs 根因，引导模型向前追溯
3. **代码执行归因**：ExecutionNode 报错时归因于提交代码的上游 Agent
4. **两个示例**：示例1（代码执行失败→代码提交者）、示例2（系统异常→消息所属 Agent）
5. **Step 约束**：step 必须是该 Agent 自己发言的 step，不同候选 step 不可重复

- outputs：最多 3 条示例 + "共N次转换"
- execution_chain：超过 20 步时用 `_compress_execution_chain` 压缩

## 五、实验结论与下一步

### 关键结论

1. **结构化推理脚手架很重要**：旧版的 assessment+root_cause_analysis 输出结构充当了 Chain-of-Thought 的作用。去掉后需要在提示词中用"从症状追溯根因"的分析原则来弥补。
2. **更多信息 ≠ 更好效果**：完整还原工作流架构的链路和转换记录，E2（闭卷）受益但 E1（开卷）受损。对于已有正确答案的场景，简洁的架构摘要反而更有效。
3. **E1 vs E2 效果常常相反**：同一改动对开卷和闭卷场景的影响方向不同，说明两种场景可能需要不同的提示词策略。
4. **20 条采样的方差较大**：AG/110 在多个版本间反复变化，属于 LLM 的边界判断问题。需要更大样本量来稳定评估。

### 下一步建议

1. **用当前 v2 版本跑全量数据**（`--n_ag -1 --n_hc -1`），获得更稳定的指标
2. **考虑 E1/E2 使用不同提示词策略**：E2 闭卷场景可能受益于更详细的工作流架构信息
3. **跑 baseline（B1-B6）全量对比**：确认 feedback 方法在全量数据上的优势
